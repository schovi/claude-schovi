# Analysis Template Reference

This document defines the structure for problem analyses generated by `/schovi:analyze-problem`.

## Template Types

### Full Analysis
Used for complex problems requiring comprehensive exploration. Includes complete technical details, multiple solution proposals, and implementation guidance.

**When to use**: Complex features, architectural changes, unclear problems, investigations

### Quick Analysis
Used for simple problems with clear solutions. Focuses on core problem, single solution, and key references only.

**When to use**: Simple bug fixes, straightforward enhancements, well-defined tasks (use `--quick` flag)

## Field Descriptions

### YAML Frontmatter

- **jira_id** (optional): Jira issue key (EC-1234, IS-8046, etc.)
- **pr_url** (optional): GitHub PR URL if analysis is from PR
- **issue_url** (optional): GitHub issue URL if analysis is from issue
- **title** (required): Brief problem description (50-80 chars)
- **problem_type** (required): Category of problem (bug|feature|investigation|performance|refactor)
- **severity** (required): Impact level (critical|high|medium|low)
- **created_date** (required): Date analysis was generated (YYYY-MM-DD)
- **created_by** (optional): Email of analyst if available

### Sections

#### Problem Summary (Full & Quick)
**Purpose**: Clearly state what the problem is and why it matters

**Core Issue**: Essential problem in 1-2 sentences
**Impact**: Who/what is affected and measurable impact
**Severity**: Classification with justification
**Urgency**: Timeline consideration (full only)
**Context**: Relevant background (full only)

#### Current State Analysis (Full Only)
**Purpose**: Document technical understanding of the problem space

**Affected Components**: List of files/modules involved with roles
**Flow Analysis**: User flow and/or data flow showing problem occurrence
**Dependencies**: Direct, indirect, and integration dependencies (conditional)
**Issues Identified**: Specific problems found with evidence and root causes

#### Current State (Quick Only)
**Purpose**: Brief technical snapshot

**Affected Components**: Key files with current behavior
**Root Cause**: Quick explanation of what's wrong

#### Solution Proposals (Full Only)
**Purpose**: Present 2-3 viable solutions with trade-off analysis

For each option:
- **Approach**: High-level description
- **Key Changes**: Specific modifications with file references
- **Pros**: Benefits and advantages
- **Cons/Trade-offs**: Limitations and concerns
- **Effort Estimate**: Time/complexity sizing
- **Risk Level**: Risk assessment with reasoning

Mark recommended option with ‚≠ê

#### Solution (Quick Only)
**Purpose**: Single clear solution path

**Approach**: Brief description
**Changes Needed**: Checklist of actions
**Pros/Cons**: Key trade-offs
**Effort/Risk**: Quick assessment

#### Implementation Guidance (Full & Quick)
**Purpose**: Provide actionable direction for execution

**Recommended Approach**: Choice and rationale (full only)
**Key Considerations**: Technical points to remember (full only)
**Testing Requirements**: What to test and how
**Tests to Update/Create**: Specific test files and scenarios
**Deployment & Rollout**: Strategy for risky changes (full, conditional)

#### Resources & References
**Purpose**: Link to relevant code, issues, docs, and stakeholders

**Code Locations**: Primary and related files with descriptions
**Related Issues/PRs**: Links to Jira, GitHub, etc.
**Documentation**: Architecture, API, wiki links
**Stakeholders**: Who to involve (conditional)
**Historical Context**: Past decisions or attempts (conditional)

## Writing Guidelines

### Be Specific
‚ùå "The validation is broken"
‚úÖ "Field validation in `FieldValidator.ts:67` rejects valid email addresses with '+' characters"

### Quantify Impact
‚ùå "This affects many users"
‚úÖ "This affects ~500 users daily (15% of active users) who cannot save their settings"

### Provide Evidence
‚ùå "The code is slow"
‚úÖ "API response time is 2.3s (p95) vs target 200ms, traced to N+1 query in `UserService.ts:123`"

### Use File References
‚ùå "Update the controller"
‚úÖ "Update `api/controllers/MappingController.ts:123`"

### Make Trade-offs Clear
‚ùå "Option 1 is better"
‚úÖ "Option 1 is faster to implement (2 days vs 1 week) but increases technical debt vs Option 2's cleaner architecture"

### Be Actionable
‚ùå "Consider performance"
‚úÖ "Add Redis caching layer in `CacheService.ts` to reduce database queries by ~80%"

## Severity Levels

- **Critical**: System down, data loss, security breach, blocking all users
- **High**: Major functionality broken, blocking many users, significant revenue impact
- **Medium**: Feature degraded, workaround exists, affecting subset of users
- **Low**: Minor issue, cosmetic problem, low user impact

---

## Problem Types

- **bug**: Existing functionality not working as intended
- **feature**: New capability to build
- **investigation**: Unclear problem needing exploration
- **performance**: Speed, efficiency, or resource usage issue
- **refactor**: Code quality/architecture improvement
